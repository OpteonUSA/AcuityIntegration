Acuity Integration - Feasibility Analysis

JaroDesk → Acuity (ClearValue) → JaroDesk  |  PDR / PDC Products
Prepared: April 17, 2026  |  Status: Exploratory  |  Source: AcuityExchange Framework Spec

Bottom Line

  • This integration is feasible — Acuity's XML protocol closely mirrors our existing Magellan (AVM) integration
  • Fits under AlternativeProductsRouter as a new child flow alongside AVM and Flood Cert
  • One key difference: Acuity is asynchronous (results return hours/days later), requiring a two-flow design
  • ~70% of the work is reusable from existing JaroDesk patterns (auth, file upload, tagging, delivery)
  • Authentication is simpler than our current integrations (HTTP Basic Auth, no token refresh)

What Is Acuity?

AcuityExchange (by ClearValue Consulting) is a mortgage services exchange platform for ordering and receiving appraisals, inspections, title, flood, and other valuation products electronically. It uses an XML message-based protocol over HTTPS with ~23 message types covering the full order lifecycle from placement through final delivery.

The likely Opteon use case is ordering PDR (Property Data Report) and/or PDC (Property Data Collection) products — alternative, lighter-weight inspection/data collection products (possibly branded as "Valor").

Key Difference: Asynchronous Delivery Our existing integrations (Magellan, AFR) return results in the same flow run. Acuity sends an immediate acknowledgement, then delivers the completed report hours or days later via a separate HTTP POST. This requires a two-flow design.

Proposed Architecture: Two-Flow Design

AlternativeProductsRouter (parent) | +-- Magellan AVM child (sync - XML - result in ~5 min) +-- AFR Flood Cert child (sync - JSON - result instant) +-- Acuity PDR/PDC child (async - XML - result hours/days later) | +-- Outbound child flow: called by router, places order, returns acknowledgement +-- Inbound standalone flow: HTTP trigger, receives AcuityReport, uploads to JaroDesk

Flow 1: Outbound (JaroDesk → Acuity)

  • Triggered by JaroDesk webhook (via router)
  • Gets order details from JaroDesk API
  • Maps fields to AcuityOrder XML
  • POSTs to Acuity with Basic Auth
  • Parses AcuityAcknowledgement
  • Logs order mapping for later lookup
  • Returns to router (does NOT wait for result)

Flow 2: Inbound (Acuity → JaroDesk)

  • Triggered by Acuity HTTP POST to our endpoint
  • Parses XML message, identifies message type
  • On AcuityReport: extracts Base64 document
  • Looks up JaroDesk order via stored mapping
  • Uploads PDF to JaroDesk (3-step pattern)
  • Tags order & triggers delivery
  • Logs completion, notifies team via Teams
Why not one flow? The router can't "wait" for Acuity like it waits for Magellan/AFR. The outbound flow fires and forgets. When Acuity finishes (potentially days later), the inbound flow independently completes the job.

Authentication

Acuity uses HTTP Basic Auth — significantly simpler than our existing vendor integrations:

Authorization: Basic [Base64(username:password)]
  • No token refresh logic needed (unlike JaroDesk's rolling session tokens)
  • No separate login endpoint (unlike Reggora's dual-URL auth)
  • Credentials stored in Power Automate environment variables (DEV vs PROD)
  • TLS 1.2 required (standard for Power Automate HTTP connector)

Build vs. Reuse Breakdown

Reusable from Existing Environment (~70%)

  • JaroDesk authentication (Dataverse TokenCache)
  • JaroDesk order retrieval (GET /v1/order/{id})
  • 3-step file upload (signed URL → S3 → attach)
  • Order tagging & delivery workflow
  • SharePoint logging pattern
  • Teams notification pattern
  • AlternativeProductsRouter dispatch logic
  • Environment variable pattern (DEV/PROD)

New Build Required (~30%)

  • Outbound flow (XML construction & POST)
  • Inbound flow (HTTP trigger & XML parsing)
  • Product code lookup table
  • Order mapping storage (Jaro ID ↔ Acuity ref)
  • New routing condition in AlternativeProductsRouter
  • Micro integration docs for Acuity endpoints
